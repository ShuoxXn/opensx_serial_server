# 烧录指南

本文档说明如何把当前单型号固件烧录到 ESP32 设备。

## 准备

1. 确认 ESP-IDF 环境已加载。
2. 使用 USB 串口连接 ESP32 设备。
3. 确认当前目录是项目根目录。

查看可用串口：

```bash
ls /dev/ttyUSB* /dev/ttyACM* 2>/dev/null
```

常见串口路径：

| 系统 | 示例 |
| --- | --- |
| Linux | `/dev/ttyUSB0`、`/dev/ttyACM0` |
| macOS | `/dev/cu.usbserial-*` |
| Windows | `COM3`、`COM4` |

## 构建并烧录

推荐直接执行：

```bash
idf.py build flash monitor
```

指定串口：

```bash
idf.py -p /dev/ttyUSB0 build flash monitor
```

只烧录，不进入监视器：

```bash
idf.py -p /dev/ttyUSB0 flash
```

只打开串口监视器：

```bash
idf.py -p /dev/ttyUSB0 monitor
```

退出监视器：

```text
Ctrl+]
```

## 擦除 Flash

如果设备配置混乱，或分区表变更后启动异常，可以完整擦除后重新烧录：

```bash
idf.py -p /dev/ttyUSB0 erase-flash
idf.py -p /dev/ttyUSB0 flash monitor
```

## 手动 esptool 烧录

一般开发不需要手动调用 `esptool.py`。如果需要使用已构建产物手动烧录，可以参考：

```bash
esptool.py --chip esp32 \
  --port /dev/ttyUSB0 \
  --baud 921600 \
  --before default_reset \
  --after hard_reset \
  write_flash -z \
  --flash_mode dio \
  --flash_freq 40m \
  --flash_size detect \
  0x1000 build/bootloader/bootloader.bin \
  0x8000 build/partition_table/partition-table.bin \
  0x19000 build/ota_data_initial.bin \
  0x20000 build/SP501W.bin
```

这些地址来自当前 `partitions.csv`。如果后续修改分区表，应以 `idf.py build` 输出的烧录命令或新的 `partitions.csv` 为准。

## 首次启动

烧录完成后设备会自动重启。启动日志中会打印：

```text
SHUOXIN-IOT Serial Server v...
Build time: ...
```

如果设备处于 AP 配置模式，可连接设备热点后访问 Web 页面进行配置。

## 恢复出厂

固件启动时会检测按键状态。长按按键启动时，会擦除 NVS 配置并重启。

恢复出厂会清除网络、协议、工作模式等运行配置，但不会修改固件本身。

## 常见问题

### 找不到串口

Linux 下检查设备：

```bash
ls /dev/ttyUSB* /dev/ttyACM*
```

如果权限不足，可以临时授权：

```bash
sudo chmod 666 /dev/ttyUSB0
```

长期建议把当前用户加入 `dialout` 组。

### 烧录失败

常见原因：

- 串口被其他程序占用。
- 波特率过高。
- 设备没有进入下载模式。
- USB 线只支持供电不支持数据。

可以降低烧录波特率：

```bash
idf.py -p /dev/ttyUSB0 -b 115200 flash
```

### 烧录后无法启动

建议：

1. 执行 `erase-flash` 后重新烧录。
2. 确认 `partitions.csv` 和 `sdkconfig` 匹配。
3. 查看 `idf.py monitor` 中的启动日志。
4. 检查供电和串口接线。
